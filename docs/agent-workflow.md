# 에이전트 작업 워크플로 규칙

이 문서는 저장소에서 코드 작업을 수행하는 에이전트가 따라야 할 상세 운영 규칙이다. `AGENTS.md`는 짧은 진입점으로 유지하고, 상세 정책은 이 문서에 둔다.

## 1. 작업 분해

구현 작업은 가능한 한 태스크 단위로 나눈다.

권장 분해 순서:

1. 계획/규칙 정리
2. 스캐폴딩/환경 구성
3. 핵심 도메인 로직
4. UI/통합
5. 테스트/문서
6. 정리/deslop

각 태스크는 독립적으로 설명 가능한 변경이어야 하며, 커밋도 가능하면 이 단위를 따른다.

## 2. 커밋 원칙

- 코드 변경 완료 후 `git status --short`로 변경 범위를 확인한다.
- 태스크 단위로 커밋한다.
- 커밋 메시지는 `AGENTS.md`의 Lore Commit Protocol을 따른다.
- 커밋 메시지에는 가능한 한 다음 항목을 포함한다.
  - `Constraint:` 변경을 제한한 조건
  - `Confidence:` 확신도
  - `Scope-risk:` 영향 범위
  - `Directive:` 후속 작업자가 지켜야 할 주의점
  - `Tested:` 실제 실행한 검증
  - `Not-tested:` 검증하지 못한 항목

## 3. 푸시 원칙

- 사용자가 완료형 구현을 요청했거나 푸시까지 기대하는 흐름이면, 커밋 후 원격 브랜치 상태를 확인하고 `git push`를 시도한다.
- 푸시는 원격 저장소를 변경하는 외부 작업이므로 인증, 권한, 네트워크 오류가 발생하면 즉시 근거를 보고한다.
- 푸시 전에는 다음을 확인한다.
  - `git status --short`
  - 현재 브랜치
  - 원격 저장소 URL
  - 최근 커밋 목록

## 4. 검증 원칙

구현 완료 전 다음 명령을 우선 실행한다.

```bash
npm test
npm run typecheck
npm run lint
npm run build
```

프로젝트 스크립트가 다르면 같은 목적의 명령으로 대체하고 최종 보고서에 명시한다.

## 5. 최종 보고 형식

최종 보고에는 다음을 포함한다.

- 변경 파일 요약
- 태스크별 커밋 해시
- 푸시 결과
- 검증 명령 결과
- 미커밋 변경 여부
- 남은 리스크

## 6. AGENTS.md와 docs의 역할 분리

- `AGENTS.md`에는 에이전트가 반드시 알아야 할 짧은 규칙과 문서 참조만 둔다.
- 상세한 작업 절차, 검증 목록, 커밋/푸시 정책은 `docs/`에 둔다.
- `docs/`가 존재하면 코드 작업 전에 모든 Markdown 문서를 읽고 규칙을 반영한다.
